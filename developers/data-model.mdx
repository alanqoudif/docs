## Source

The practical source is the migrations inside `atlas-swarm-core-1/supabase/migrations` and the generated types in `src/integrations/supabase/types.ts`.

## Product tables

- `profiles`: user data and preferences such as `custom_instructions`.
- `conversations`: user projects, which appear in the `/chat/$projectId` route.
- `messages`: user and assistant messages inside each conversation.
- `artifacts`: current outputs linked to a conversation.
- `artifact_versions`: output versions after creation and editing.
- `websites`: multi-page websites created through `create_website`.
- `website_pages`: website pages published at `/sites/{slug}`.
- `presentations`: presentation data.
- `skills`: instructions that can be injected into chat and classification context.
- `ai_usage_log`: AI model usage and cost tracking.

## Auth and helper tables

- `phone_otps`: phone OTP codes.
- `desktop_handoff_codes`: short-lived codes for linking the desktop app.
- `affiliate_interest`: partner or affiliate interest.

## Email tables

- `email_send_log`
- `email_send_state`
- `email_unsubscribe_tokens`
- `suppressed_emails`

There are also email RPCs such as enqueue/read batch, depending on the migrations.

## Security notes

Every API that changes user data must verify the Bearer JWT through `requireAuth`, then verify resource ownership, especially for `conversationId` and `artifactId`. Agent tools use the admin client, so scoping by user and conversation is essential, not optional.
