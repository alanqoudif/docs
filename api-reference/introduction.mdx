<Note>
The public Mibyan API is being prepared. This page explains what API access is intended for and what information to provide if you need an integration.
</Note>

## Who needs API access?

Most users should use the Mibyan web application directly. API access is for teams that want to connect Mibyan to another product, internal system, automation, or workflow.

Common use cases include:

- Generating structured business documents from another app.
- Creating draft reports from form submissions.
- Producing pitch deck drafts from CRM or onboarding data.
- Connecting Mibyan outputs to an internal review workflow.
- Automating recurring business analysis or summaries.

## Current status

The stable public API is not yet published as a self-serve SDK. Until public access is available, integrations should be discussed with the Mibyan team.

When the API is ready, this documentation should include:

- Authentication and access setup.
- Request and response examples.
- Supported output types.
- Rate limits and usage expectations.
- Error handling.
- Webhooks or callback patterns if supported.
- Security and data handling guidelines.

## What to prepare before requesting access

If you need API access, prepare:

- Your company or product name.
- The system you want to connect.
- The output types you need: documents, decks, spreadsheets, pages, or websites.
- Expected request volume.
- Whether outputs contain sensitive data.
- Required languages.
- Export requirements.
- Any compliance or data residency constraints.

## Integration planning checklist

Before building an integration, clarify:

- Who triggers the request?
- What data is sent to Mibyan?
- What output should be returned?
- Who reviews the output before it is used?
- Where should the final output be stored?
- What happens if the output needs revision?
- What information must never be sent?
